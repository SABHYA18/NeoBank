NeoBank Super-App — Technical Architecture & Operational Guide

This technical architectural whitepaper outlines the systematic design decisions, internal framework operations, and constraints mapping out the NeoBank Super-App platform ecosystem.

🏦 Part 1: The Backend (Spring Boot Monolith)

The core layer is packaged as a high-performance Modular Monolith running on Java 21 and Spring Boot 3.x architecture, enforcing rigid isolation interfaces between discrete business boundaries.

1. Security & Authentication Layer com.neobank.auth

Stateless JJWT Authentication: Identity confirmation utilizes completely stateless processing structures. Upon successful validation of credentials, the server engine yields a JSON Web Token (JWT) stamped using SHA cryptographic key rules. The customer interface attaches this record inside the Authorization: Bearer <token> call headers for subsequent validation sweeps.

Spring Security Filters: A custom filter element named JwtAuthFilter screens each inbound packet, decrypts token payloads, ensures validation windows remain open, and populates the UserPrincipal model context inside Spring's active SecurityContextHolder matrix.

2. Core Banking & Ledger Subsystem com.neobank.banking

Double-Entry Ledger Accounting: System auditing guarantees immutable accounting entries. Financial adjustments are computed dynamically by pulling aggregations directly out of the ledger_entries ledger history tables. Individual balances are never modified as raw, mutable column fields.

Idempotency Safeguard Filters: Prevent duplication vulnerabilities triggered by connection retries by enforcing strict verification parameters based on incoming client-supplied idempotencyKey strings. Duplicate requests match previous logs in the database and return cached tracking blocks instantly.

JPA Optimistic Concurrency Locking: The account structural entity model uses an internal @Version concurrency counter field. Concurrent write collisions cause Hibernate to issue an ObjectOptimisticLockingFailureException error event, which prompts immediate rollback sequences to maintain systemic isolation ceilings.

3. PayFlow Core Wallet & P2P Subsystem com.neobank.payflow

Immutable Single Wallet Relationship: Each distinct profile domain mapping is limited to exactly one digital wallet container entity. During initial setup calls via POST /api/v1/payflow/wallet/initialize, the target client permanently attaches an operational bank account link preference. This structural mapping cannot be altered or re-bound later.

The ₹1,000 (1k) Floor Safety Limit Mechanics:
• Activation Ceiling: Setup initialization sequences require pulling a base startup resource deposit sizing equal to or exceeding ₹1,000.00 straight from the connected banking instrument.
• Floor Constraints: Core wallet table check limits prevent internal balances from slipping beneath the absolute ₹1,000.00 minimum floor during peer shifts or request settlements.

Isolated Balances & Pull Recharges: System database scopes remain completely isolated. The wallet model touches base with core banking records only during structured recharge procedures triggered at POST /api/v1/payflow/wallet/recharge. Pure peer-to-peer (P2P) transfers operate inside isolated payflow_wallets schemas.

🎨 Part 2: The Frontend (React + TS + Tailwind v4)

The presentation ecosystem is designed as an optimized Single Page Application utilizing Vite 5 compiler engines, React 18 framework rules, TypeScript code structures, and styled via **Tailwind CSS v4** layout sheets.

1. Aesthetic Design System index.css

Apple-Inspired Typography & Spacing: Embeds the modern, clean Outfit typeface featuring micro-thin layout weights, extra character spacing tracking metrics, and sharp visual grid borders mapped at border-[0.5px] border-zinc-200/60.

Micro-Animations & Visual Anchors: Content modules elevate cleanly via hover:-translate-y-0.5 physics rules while rendering subtle background box glow shadows. Alert prompts slide onto screens from the left margin, nesting at the top-left corner (fixed top-6 left-6 z-[99999]) using dark contrast plates to preserve crisp layer separation guidelines over lower visual structures.

2. Unified Workspace Portals App.tsx

  • Overview Portal Dashboard: Renders complete aggregate valuation metrics using thin style typography weights coupled with real-time reactive card components showing linked check assets. Features inline tools to generate new operational accounts easily.
  • Transactions Workspace: Forms handling cash inflows, cash points execution, and inside asset splits. Outputs a live code block mapping the current client-side generated Idempotency key context tracker.
  • Ledger Workspace Audit Logs: Clean structural accounting sheets tabulating double-entry logs, compiling clear credit/debit transaction paths, execution reference codes, and updated balance metrics.
  • PayFlow Wallet Workspace Portal: Renders an animated virtual card component displaying balance configurations, user strings, and connected ledger pathways. Handles linking workflows (enforcing the base ₹1,000 activation parameters), instant peer transfers, and provides an actionable requests activity dashboard panel.

3. Automated Security Token Handshake Sequence

Toggling the **"Live Mode"** control switch widget triggers an immediate operational profile migration from localized state simulations to the active backend runtime processing on port 8081. This zero-configuration automated pipeline runs the following background sequence sequence:

1. POST /api/v1/auth/signup  --> Registers default credentials (jane_doe / Password123!)
2. POST /api/v1/auth/login   --> Secures signed stateless JJWT accessToken string
3. State Interception Hook  --> Saves bearer token in state and appends Authorization headers
4. System Refresh            --> Synchronizes all operational account values instantly from the database
